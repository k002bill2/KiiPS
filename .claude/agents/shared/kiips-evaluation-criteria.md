# KiiPS 평가 기준표 (Evaluation Criteria)

> Generator와 Evaluator가 공유하는 KiiPS 페이지 품질 기준.
> Harness Engineering 패턴 기반, KiiPS 엔터프라이즈 환경에 적응.

---

## 채점 항목

### 1. 패턴 준수 (비중: 40%)

KiiPS 표준 패턴을 정확히 따르는가?

| 등급 | 점수 | 기준 |
|------|------|------|
| 합격 | 8-10 | 표준 JSP 구조, Include 체계(inc_filter_main, inc_main_button, inc_grid_main), RealGrid 초기화 패턴, MainComponent/Constant 분리 |
| 보통 | 5-7 | 구조는 맞으나 세부 패턴 누락 (Lookup 초기화, columnGroup 미설정 등) |
| 불합격 | 0-4 | 비표준 구조, Include 미사용, 인라인 이벤트 핸들러, jQuery 대신 vanilla JS |

**세부 체크리스트**:
- [ ] JSP Tiles 레이아웃 사용 (`tiles-def` 정의)
- [ ] Include 3종 세트: `inc_filter_main.jsp`, `inc_main_button.jsp`, `inc_grid_main.jsp`
- [ ] RealGrid: `createMainGrid()` + `GridView`/`DataProvider` 분리
- [ ] AJAX: `logosAjax` 패턴 사용 (raw $.ajax 금지)
- [ ] 검색조건: `MainComponent` + `Constant` + `SEARCH_CONDITION` 빌더
- [ ] 버튼: `inc_main_button.jsp`에 도메인별 등록
- [ ] 이벤트: `$(document).on()` 위임 패턴

### 2. 기능 완성도 (비중: 30%)

SPEC에 명시된 기능이 모두 동작하는가?

| 등급 | 점수 | 기준 |
|------|------|------|
| 합격 | 8-10 | CRUD 전체 동작, 검색 필터 연동, 페이징 정상, 엑셀 다운로드 |
| 보통 | 5-7 | 기본 조회/등록 동작, 일부 기능 미구현 (엑셀, 상세 필터) |
| 불합격 | 0-4 | 조회 불가, AJAX 에러, 그리드 미렌더링 |

**세부 체크리스트**:
- [ ] 목록 조회 (검색조건 반영)
- [ ] 신규 등록 (모달 → 저장 → 목록 갱신)
- [ ] 수정 (행 선택 → 모달 → 저장)
- [ ] 삭제 (확인 → 처리 → 목록 갱신)
- [ ] 페이징 (RealGrid paging 또는 서버 페이징)
- [ ] 엑셀 다운로드 (`exportGrid` 패턴)
- [ ] 검색 초기화 (btn_reload)

### 3. 접근성/반응형 (비중: 15%)

WCAG 2.1 AA + Bootstrap 반응형 기준을 충족하는가?

| 등급 | 점수 | 기준 |
|------|------|------|
| 합격 | 8-10 | ARIA 라벨 완비, 키보드 탐색 가능, 반응형 정상, 다크테마 지원 |
| 보통 | 5-7 | 기본 접근성 존재, 일부 ARIA 누락, 반응형 부분 대응 |
| 불합격 | 0-4 | ARIA 전무, 모바일 깨짐, 다크테마 미대응 |

**세부 체크리스트**:
- [ ] `<input>`, `<button>`에 `aria-label` 존재
- [ ] `<img>`에 `alt` 속성
- [ ] 다크테마: `[data-theme=dark]` 셀렉터 사용
- [ ] 터치 타겟 ≥ 44px
- [ ] 가로 스크롤 미발생

### 4. 보안 (비중: 15%)

KiiPS 보안 기준을 충족하는가?

| 등급 | 점수 | 기준 |
|------|------|------|
| 합격 | 8-10 | MyBatis `#{}` 전용, Controller 입력 검증, XSS 방어, CSRF 토큰 |
| 보통 | 5-7 | 기본 보안 존재, 일부 검증 누락 |
| 불합격 | 0-4 | `${}` 사용, 입력 미검증, XSS 취약 |

**즉시 불합격 트리거** (점수 무관):
- MyBatis `${}` 바인딩 사용
- Controller에서 입력 검증 누락
- 하드코딩된 비밀번호/API 키
- `$.html()` + 사용자 입력 (XSS)

## 판정 기준

### 가중 점수 계산

```
가중 점수 = (패턴 준수 × 0.4) + (기능 완성도 × 0.3) + (접근성 × 0.15) + (보안 × 0.15)
```

### 판정 등급

| 판정 | 조건 | 후속 조치 |
|------|------|----------|
| **합격** | 가중 ≥ 7.0 AND 즉시불합격 없음 | 완료 보고 |
| **조건부 합격** | 가중 5.0~6.9 | 피드백 반영 후 재검수 |
| **불합격** | 가중 < 5.0 OR 패턴준수 ≤ 4 OR 보안 ≤ 4 | 재생성 또는 근본 재설계 |

### 특별 규칙
- 패턴 준수 4점 이하 → **무조건 불합격** (KiiPS 표준 위반)
- 보안 4점 이하 → **무조건 불합격** (보안 취약)
- 즉시 불합격 트리거 1개 이상 → **무조건 불합격**

## KiiPS 안티패턴 금지 목록 (Anti-Slop)

> "AI Slop"의 KiiPS 버전 — Generator가 빠지기 쉬운 뻔한 실수

### 절대 금지

| 안티패턴 | 올바른 패턴 | 위반 시 |
|----------|-----------|---------|
| 인라인 `style=""` 속성 | CSS 클래스 사용 | 패턴 감점 |
| `<input type="checkbox">` 직접 사용 | `checkbox-custom` 클래스 패턴 | 패턴 감점 |
| RealGrid `styles.textAlignment: "center"` | 기본 중앙정렬이므로 생략 | 패턴 감점 |
| raw `$.ajax()` 호출 | `logosAjax` 래퍼 사용 | 패턴 감점 |
| `onclick=""` 인라인 이벤트 | `$(document).on("click", ...)` | 패턴 감점 |
| `.dark`, `.theme-dark` 셀렉터 | `[data-theme=dark]` 전용 | 접근성 감점 |
| `var` 선언 (ES6) | JSP 내 `var` 허용 (Java 8 환경) | 감점 없음 |
| 날짜 입력 직접 구현 | `flatpickr-basic` 클래스 사용 | 패턴 감점 |
| `System.out.println` | SLF4J 로거 사용 | 보안 감점 |
| MyBatis `${}` | `#{}` 바인딩 전용 | **즉시 불합격** |

## 피드백 형식 (필수)

Evaluator는 반드시 다음 형식으로 피드백을 작성합니다:

```markdown
## QA Report

**전체 판정**: [합격 / 조건부 합격 / 불합격]
**가중 점수**: X.X / 10.0
**라운드**: R[N] (최대 R3)

### 항목별 점수

| 항목 | 점수 | 코멘트 |
|------|------|--------|
| 패턴 준수 (40%) | X/10 | [1줄] |
| 기능 완성도 (30%) | X/10 | [1줄] |
| 접근성/반응형 (15%) | X/10 | [1줄] |
| 보안 (15%) | X/10 | [1줄] |

### 구체적 개선 지시

1. **[WHERE]** `파일명:라인` — **[WHY]** 규칙 근거 — **[HOW]** 구체적 수정 방법
2. **[WHERE]** `파일명:라인` — **[WHY]** 규칙 근거 — **[HOW]** 구체적 수정 방법

### 안티패턴 발견

- [있음/없음] — 발견 시 목록 나열

### 방향 판단

[현재 방향 유지] 또는 [완전히 다른 접근 시도]
(사유: ...)
```

### 피드백 3요소 규칙

모든 개선 지시는 반드시 3요소를 포함:

| 요소 | 설명 | 나쁜 예 | 좋은 예 |
|------|------|---------|---------|
| **WHERE** | 어디가 문제 | "검색이 안 됩니다" | `inc_filter_main.jsp:23` |
| **WHY** | 왜 문제 (규칙 근거) | "패턴에 맞지 않습니다" | "SEARCH_CONDITION 빌더 미사용 (kiips-search-filter-guide 위반)" |
| **HOW** | 어떻게 수정 | "수정해주세요" | "`MainComponent.addCondition('FUND_NM', 'text', '펀드명')` 추가" |

## 반복 검수 규칙

| 라운드 | 규칙 |
|--------|------|
| R1 | 전체 항목 채점 + 피드백 |
| R2 | R1 피드백 반영 확인 + 기존 합격 항목 퇴보 여부 + 새 이슈 |
| R3 (최종) | 동일 항목 3회 연속 불합격 → "구현 방식 자체 변경" 지시 |
| R3 이후 | 현재 상태로 전달 + 미해결 이슈 보고 |
