# 증거 기반 완료 규칙 (Evidence-Based Completion)

> Claude Forge Golden Principle #10을 KiiPS 환경에 맞게 적응.
> "아마 될 것"은 "아마 무죄"와 같습니다 — 전혀 유효하지 않습니다.

## 철칙

```
실행 증거 없이 완료를 선언하지 마십시오.
```

이 메시지에서 검증 명령을 실행하지 않았다면, 통과했다고 주장할 수 없습니다.
이전 실행은 유효하지 않습니다 — 그 이후 코드가 변경되었을 수 있습니다.

## 검증 게이트 함수

완료, 성공, 또는 만족에 대한 모든 주장 전에:

```
1. IDENTIFY  — 이 주장을 증명하는 명령은 무엇인가?
2. RUN       — 전체 명령을 실행 (새로, 완전하게)
3. READ      — 전체 출력을 읽고, exit code 확인, 실패 수 확인
4. VERIFY    — 출력이 주장을 확인하는가?
               아니오 → 증거와 함께 실제 상태 보고
               예     → 증거와 함께 결과 선언
5. CLAIM     — 이제서야 결과를 진술

어떤 단계든 건너뛰기 = 검증이 아니라 추측.
```

## KiiPS 검증 체크리스트

| 주장 | 필수 증거 | 불충분한 증거 |
|------|----------|-------------|
| 빌드 성공 | `mvn package` 출력: BUILD SUCCESS | "컴파일 됐으니 빌드도 될 것" |
| 테스트 통과 | `mvn test` 출력: Tests run: N, Failures: 0 | 이전 실행, "통과할 것" |
| 서비스 실행 중 | `curl` 또는 `lsof -i :포트` 응답 확인 | "start.sh 실행했으니 될 것" |
| SCSS 컴파일 | SCSS 컴파일러 출력: 0 errors | "구문이 올바르니 될 것" |
| MyBatis 안전 | `grep '${' *.xml` 결과 0건 | "#{} 으로 바꿨으니 될 것" |
| 버그 수정됨 | 회귀 테스트 통과 증거 | "코드 변경했으니 수정됨" |
| 에이전트 작업 완료 | 변경 파일 diff 확인 | 에이전트 자체 "성공" 보고 |
| 요구사항 충족 | 항목별 체크리스트 검증 | "테스트 통과했으니 요구사항 충족" |
| 다크 테마 적용 | [data-theme=dark] 셀렉터 확인 | "SCSS 수정했으니 될 것" |

## 금지 표현 (Red Flags)

다음 표현이 떠오르면, 검증을 실행하세요:

| 생각 | 현실 |
|------|------|
| "아마 될 것" | 검증을 실행하세요 |
| "확신합니다" | 확신은 증거가 아닙니다 |
| "이번만 예외" | 예외는 없습니다 |
| "린트 통과했으니..." | 린트 ≠ 컴파일러 ≠ 테스트 |
| "에이전트가 성공했다고 했으니" | 독립적으로 검증하세요 |
| "단순한 변경이라 검증 불필요" | 단순한 변경도 깨집니다 |
| "아까 실행했으니" | 아까 ≠ 지금. 다시 실행하세요 |
| "아마", "될 것", "~인 것 같다" | 확률적 언어는 증거가 아닙니다 |
| "좋습니다!", "완벽!", "완료!" | 검증 전에 만족을 표현하지 마세요 |

## 적용 시점

**항상**, 다음을 하기 전에:
- 성공 또는 완료 선언
- 결과에 대한 만족 표현
- 작업 상태에 대한 긍정적 진술
- 커밋 또는 PR 생성
- 작업을 완료로 표시
- 다음 작업으로 이동
- 하위 에이전트에게 위임

## 핵심

검증에 지름길은 없습니다.

명령을 실행하세요. 출력을 읽으세요. 그리고 — 그때서야 — 결과를 진술하세요.

이것은 협상 불가입니다.
