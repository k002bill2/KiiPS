# /plan - 구조화된 작업 계획

> Forge Investigation Protocol(인터뷰→탐색→설계→검증)을 KiiPS에 적응
> 기존 `kiips-feature-planner` 스킬과의 차이: planner는 대규모 Feature 계획, `/plan`은 세션 내 작업 계획

$ARGUMENTS 작업에 대한 5단계 구조화된 계획을 수립하세요.

---

## 5단계 프로토콜

### 1. CLARIFY (인터뷰) - 사용자에게 질문

다음을 명확히 확인하세요:

- **변경 범위**: 어떤 모듈? (FD/IL/AC/SY/LP/EL/UI/COMMON/UTILS/BATCH)
- **영향 범위**: DB 변경? API 변경? UI 변경?
- **제약사항**: 기존 데이터 호환? 다른 화면 영향?
- **우선순위**: 기능 완성 vs 품질 vs 성능?

불명확한 부분은 **반드시** 사용자에게 질문하세요. 가정하지 마세요.

### 2. EXPLORE (탐색) - 코드베이스 조사

Explore 에이전트 또는 Grep/Glob 활용:

- **유사 화면/기능 탐색**: 기존 패턴 확인 (같은 도메인의 유사 JSP/Controller/DAO)
- **의존성 체인 추적**: Controller → Service → DAO → Mapper → Table
- **공통 모듈 사용 확인**: COMMON/UTILS에서 재사용 가능한 코드
- **참조 관계**: 이 코드를 사용하는 다른 코드 확인

### 3. PLAN (설계) - 변경 계획 수립

출력 형식:

```
## 변경 계획

### 변경 파일 목록 (HARD-GATE: 3+ 파일이면 승인 필요)
| # | 파일 | 변경 유형 | 변경 내용 |
|---|------|----------|----------|
| 1 | ... | 신규/수정 | ... |

### 위험 평가
- [ ] DB 스키마 변경: 예/아니오
- [ ] COMMON/UTILS 변경: 예/아니오
- [ ] API 인터페이스 변경: 예/아니오
- [ ] 다른 화면 영향: 예/아니오

### 구현 순서
1. ...
2. ...
```

### 4. VALIDATE (사전 검증) - 계획 검증

실행 전 체크:

- **빌드 영향**: 어떤 Maven 모듈을 빌드해야 하는가?
- **보안 체크**: #{} vs ${} 바인딩, XSS 노출 가능성
- **기존 패턴 준수**: KiiPS 코딩 컨벤션 위반 여부
- **회귀 위험**: 기존 기능에 영향 없는지

### 5. EXECUTE (승인 후 실행) - 구현

사용자 승인 후:

- 계획된 순서대로 구현
- 중간 체크포인트 설정 (3개 파일마다)
- 각 단계별 검증 (→ [verification.md](./verification.md) 참조)
- 완료 후 전체 빌드 검증

---

## 위험도 매트릭스

| | 낮은 복잡도 | 높은 복잡도 |
|---|-----------|-----------|
| **낮은 영향** | 바로 실행 | 계획 후 실행 |
| **높은 영향** | 계획 후 실행 | 상세 계획 + 단계별 승인 |

**높은 영향 기준**: DB 변경, COMMON/UTILS 변경, API 인터페이스 변경, 5+ 파일 변경
