# Diagnose - 진단 우선 디버깅

에러나 문제를 해결할 때, **코드를 먼저 변경하지 않고** 진단 보고서를 먼저 작성합니다.

**Usage:** `/diagnose <에러 메시지 또는 증상 설명>`

**Examples:**
- `/diagnose 8088 포트에서 Connection refused 에러`
- `/diagnose 펀드 목록 조회 시 500 에러`
- `/diagnose SCSS 컴파일 후 다크테마 깨짐`

---

## Instructions for Claude

### Phase 1: 정보 수집 (READ ONLY)

이 단계에서는 **파일을 절대 수정하지 않습니다.**

1. **에러 로그 확인**
   - `logs/log.$(date "+%Y-%m-%d")-0.log` 에서 관련 에러 검색
   - 스택 트레이스 전체 확인 (일부만 보지 말 것)

2. **소스 파일 조사**
   - 에러가 발생하는 파일과 라인 번호 확인
   - 관련 클래스/메서드의 전체 컨텍스트 읽기

3. **최근 변경 확인**
   - `svn log -l 10` 으로 최근 변경 내역 확인
   - 문제 발생 시점 전후 변경 사항 파악

4. **환경 상태 확인**
   - 서비스 실행 상태, 포트 바인딩, DB 연결

### Phase 2: 진단 보고서 출력

수집된 정보를 바탕으로 아래 형식의 보고서를 출력합니다:

```
## 진단 보고서

### 증상
[관찰된 에러/증상 요약]

### 근본 원인 가설
1. [가설 1] - 신뢰도: 높음/중간/낮음
   - 근거: [로그/코드에서 발견한 증거]
2. [가설 2] - 신뢰도: ...

### 변경 대상 파일
| 파일 | 변경 내용 | 위험도 |
|------|----------|--------|
| path/to/file.java | 설명 | 낮음/중간/높음 |

### 위험 평가
- 영향 범위: [이 변경이 영향을 미치는 다른 모듈/기능]
- 되돌리기 난이도: [쉬움/보통/어려움]
```

### Phase 3: 승인 대기

**사용자가 승인할 때까지 코드를 변경하지 않습니다.**

- "진행해" / "수정해" 등의 승인을 기다립니다
- 사용자가 다른 가설을 선택할 수 있습니다
- 추가 조사가 필요하면 Phase 1로 돌아갑니다

### Phase 4: 정밀 수정

승인 후:
1. **한 파일씩** 변경 (동시에 여러 파일 변경 금지)
2. 각 변경 후 **즉시 검증** (컴파일, 로그 확인)
3. 변경이 상황을 **악화시키면 즉시 되돌리기**
4. **2회 시도 후 실패 시 중단** - 사용자에게 추가 맥락 요청

---

## Critical Rules

- **READ ONLY FIRST**: Phase 1에서 절대 파일 수정 금지
- **가설 먼저**: 코드 변경 전 반드시 가설 제시
- **최소 변경**: 필요한 최소한의 수정만 수행
- **2회 실패 시 중단**: 같은 접근법으로 2회 실패하면 멈추고 사용자에게 보고
- **관련 없는 파일 건드리지 말 것**: 디버깅 중 "이것도 고치면 좋겠다" 금지
